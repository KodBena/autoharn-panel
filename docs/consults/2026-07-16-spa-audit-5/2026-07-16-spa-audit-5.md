# Independent Audit: autoharn-panel SPA vs. the Autoharn Governance Harness

**Auditor:** independent scout session, no prior findings inherited. **Target:** the live autoharn-panel deployment at `http://localhost:8420` (backend: `uvicorn app:app --host 0.0.0.0 --port 8420 --reload`, `PANEL_READONLY=1`, schema `experience`/`experience_kernel` against Postgres host `192.168.122.1` db `toy`; frontend dev server also live on `:5173`, not separately audited since `:8420` serves the built SPA the deployment actually presents). **Method:** read `autoharn/README.md`, `docs/PROJECT-OVERVIEW.md`, `design/USER-RECIPES-FAQ.md`, and `design/Autoharn.idr` in full before touching the app; then drove the live app with headless Chromium via Playwright scripted through Bash (never the Playwright MCP tool), cross-checked every UI claim against the live Postgres database and the backend/frontend source, and checked the app's own in-repo `SPEC.md` (cited by CLAUDE.md as a binding design input) for what the SPA's own authors declared as required scope.

**A transparency note, disclosed rather than buried:** this deployment's ledger contains real-time, in-band commentary about this very audit — a `work_opened` row (id 812, slug `spa-audit-scout-5`) explicitly commissioning "a fresh, unbiased scout" to perform this exact task, and a `decision` row (id 816) recorded minutes before I began, flagging that the newly-shipped Backend Surface tab "exposes a much larger raw DB-schema surface than the rest of the app ever has, and the scout could plausibly get biased by it into recommending broader raw-schema exposure elsewhere." I note this for the record because an audit's credibility depends on saying so when the subject under audit is aware of the audit and has pre-narrated a hypothesis about it. I formed every judgment below from what I directly observed and verified (live DOM/API/DB behavior, source code, and the project's own written spec) — the Backend Surface tab is judged below purely on its own merits, and the pre-emptive note did not enter my reasoning either way.

Findings are ranked by real-world severity, missing-capability and rendering/UX defects interleaved as instructed. Each finding gives concrete repro evidence.

## CRITICAL

### 1. The obligation/dependency-tree graph — an explicitly "maintainer-demanded" P0 requirement — does not exist anywhere in the shipped app

The project's own `SPEC.md` §2.3, titled "Obligation-tree graph view," is marked **P0 (maintainer-demanded)** and is listed in §5's definition of P0 as something "the SPA is not usable without." Its description is detailed and specific: "The AND-tree from the ratified close-semantics model, drawn as a graph: leaf obligations resolve by recorded acts, interior nodes by conjunction... Color coding: red = undischarged, green = discharged, amber = ambiguous/partial... gray = superseded... Hover = synopsis card... click = item view... Live recolor on ledger-change events. Layout: layered DAG... Scale target: 500 nodes interactive." `docs/omega-observatory/2026-07-15-frontend-speed-reap.md`'s own opening line independently confirms this was part of the SPA's founding brief: "a Vue rebuild of autoharn's panel SPA (Postgres-backed append-only ledger — typed row lists, a commission-decomposition view, **a planned obligation AND-tree graph**, SSE polling ~2s)."

This is not a decorative nice-to-have. `design/Autoharn.idr` devotes its entire §2b (the obligation AND-tree, `strictBlockers`), most of §3 (the write boundary's strict-close premise, which reads that same tree), and all of §4 (composite discharge, defined as nothing but a *read* of the §2b calculus) to this one mechanism — it is the single most load-bearing derivation the kernel computes, the thing that decides whether a work item is actually allowed to close.

**What I actually saw:** live confirmation via the Backend Surface tab (and cross-checked directly against Postgres) that `work_edge_parent` (36 rows) and `work_edge_obligation` (37 rows) are real, populated, non-trivial graph structures in this deployment today — and that neither is exposed by the backend API at all ("exposed by our API: no" for both, confirmed on the Backend Surface tab). The only place any of this structure reaches the UI is `WorkItemsTab.vue`'s flat `parent_slug` column (plain mono text, not a link) and a comma-joined `blocked_by_label` string — and that component's own source comment discloses the omission plainly: *"a graph/tree view is explicitly out of scope for this item."* There is no tree, no DAG, no click-to-navigate hierarchy, no visual discharge coloring anywhere in the ten tabs I drove. A user trying to understand "why is this composite parent still open" or "what does closing this item actually unblock downstream" has no tool for it beyond manually cross-referencing plain-text slugs across rows.

### 2. The kernel's own computed, non-forgeable review-independence signal (`discharge_grade`) is exposed nowhere — while every review in this live deployment is, in fact, the two weakest grades

`design/Autoharn.idr` treats `DischargeGrade` (`SamePrincipal | SameSession | DistinctSession | DistinctDeployment`) as a first-class, *trigger-computed, never writer-supplied* fact (§R2's whole point: "a writer literally cannot supply a grade... `GradeF Draft = ()`"), verbatim-transcribed from the kernel's real `validate_independence()` function (§R9's `gradeLadder`). CLAUDE.md's own point 3 and USER-RECIPES-FAQ.md's "Review discipline" section both build directly on this: the self-declared `independence` category (technical/managerial/financial/self-review) is explicitly *not* itself proof of anything — `review_gap`'s discharge test "never looks at what a review says," and the computed grade is the one signal that can't be self-asserted.

**What I verified:** `psql` against the live `experience.review_detail` table confirms the `discharge_grade` column is real, populated, and constrained to exactly the vocabulary the Idris model describes. Querying it live: **100% of the 95 reviews in this deployment are graded `same-principal` (20) or `same-session` (75) — zero are `distinct-session` or `distinct-deployment`.** That means every one of the 75 reviews self-declared as `independence: technical` in this deployment is, per the kernel's own computed truth, merely a same-session attestation — the second-weakest possible grade — a fact the kernel itself already knows and records, and which the app never says. I confirmed via `grep` across the entire `backend/` tree that the string `discharge_grade` appears **nowhere** in the backend source; `/api/item/{id}/obligations` and `/api/review-gap` responses carry only `verdict`/`independence`/`basis`, never the grade. `ItemObligationsPanel.vue`'s review table (verified live on row 808, review 811) renders `independence: technical` as plain unstyled text — visually indistinguishable from how a genuine `distinct-deployment` review would render, were one ever to exist. This directly undercuts the one part of the domain the audit brief specifically names as needing to be "trustworthy": a reader of this SPA cannot tell a real arms-length review from a same-session self-attestation dressed in a stronger-sounding label, without independently reading free-text prose and knowing to be suspicious of it.

## SERIOUS

### 3. The Item Detail view — the app's single most-linked-to page, and a declared P0 — renders its own refs and supersession chain as dead text, not links

`SPEC.md` §2.2 ("Item view... P0") is explicit: *"refs in/out (each a hoverable link, §3)... supersede chain rendered as a chain."* The app has a genuinely well-built mechanism for exactly this (`CitationText.vue`/`CitationLink.vue`: splits any statement on `row:<id>` tokens, renders a router-link with a hover synopsis card, even Teleports the card past virtualized-list scroll-clipping to fix a real bug the code comments document finding). It is wired to the main `statement` field on every view that has one.

**What I actually saw:** it is *not* wired to the three fields that most need it. On `/item/397`'s DOM (verified via `innerHTML`), `refs (raw)` renders `row:16` as plain `<dd class="statement-text mono">` text with no anchor; `refs → rows` renders `16` as a bare `<span class="mono">`, no anchor; `supersede chain` renders `predecessors: 38 · successor: (none)` as plain mono text, no anchor. None of the three is clickable, hoverable, or even styled to suggest interactivity — while three lines above, the same page's `statement` field correctly renders its one `row:16` mention as a working hover-linked citation. Given how heavily this project's whole design (append-only ledger, `--supersedes` as "the default answer to every 'I wrote it wrong' situation," per USER-RECIPES-FAQ.md) leans on supersession chains being traceable, and given `Autoharn.idr` §5 devotes real effort to proving supersession's semantics sound, shipping the one page whose entire job is to show "everything the ledger relates to this row" with its relationship fields inert is a significant, concrete legibility gap on the app's most central view.

### 4. No full-text search anywhere in the SPA, despite the backend supporting it and the SPA's own spec naming it

`SPEC.md` §3 declares "Search (P1): server-side full-text over statements, results as 2.1 rows" as planned scope. The backend's `GET /api/rows` genuinely accepts a `q` parameter (confirmed in the endpoint map and its handler). **The frontend never uses it** — I grepped the entire `frontend/src` tree for `q` as a query parameter and found zero references; `LedgerTab.vue`'s own filter set is kind/actor/since/until/limit only, no free-text box, confirmed both in source and by driving the live UI (no search input exists on any tab).

The practical consequence is real, not hypothetical: this deployment's ledger holds 257 `decision`-kind rows, a closed vocabulary kind that (per USER-RECIPES-FAQ.md) houses several distinct governed sub-vocabularies by text-prefix convention alone — `resource:` (2 rows here, both currently `durable`-graded and therefore visible via Standing Decisions) and, per the FAQ, `taxon:`, `task-policy:`, `estimate:`, and the maintainer's own `review:`/`review-done:` queue (none populated in this deployment today, confirmed by direct `ilike` query against the live ledger — so this specific gap is prospective here, not currently denying access to real content, but it is a real, structural hole: the moment any of those rows exist, or the moment a `resource:` row is filed without a `durable` grade, there is no tab and no search that reaches it). Today, finding a specific declaration among 257 decision rows means paging through up to two full 200-row pages by eye. This is exactly the shape of gap the audit brief calls "as real a defect as a rendering bug."

## MODERATE

### 5. No `<main>` landmark, no skip link

Confirmed via DOM query on the live app: `document.querySelectorAll('main').length === 0`, and no skip-link element exists. The nav bar (correctly marked `role="tablist"` with ten `role="tab"` buttons) sits ahead of all page content in tab order on every route. A keyboard or screen-reader user must tab through all ten nav buttons (plus, on tabs with filters, several more controls) before reaching any content, on every single page load or route change — precisely the friction WCAG's "bypass blocks" guidance and `SPEC.md` §3's own "Accessibility floor (P1)" line exist to prevent.

### 6. Half-implemented ARIA tabs pattern

The nav correctly carries `role="tablist"` on the `<nav>` and `role="tab"`/`aria-selected` on each button — good instincts. But there is no `role="tabpanel"` anywhere on the content region (`document.querySelectorAll('[role=tabpanel]').length === 0`), and I confirmed live that `ArrowRight` while focused on a tab does **not** move focus or selection to the next tab (standard behavior a screen-reader user would expect once the tablist role announces). Implementing half the pattern is arguably worse than none: assistive tech that recognizes the tablist semantics will expect arrow-key behavior that silently does nothing.

### 7. A real, reproducible CSS layout bug in the Commission tab at ~600px width

At a 600px viewport, commission #741's two decomposition-item cards render inconsistently: `spa-backend-surface-view` (shorter slug) stacks cleanly — slug+badge on their own full-width line, body paragraph below. `spa-audit-scout-5` (longer slug) does not: its slug and `OPEN` badge end up in a narrow left column while the multi-line body paragraph flows to their right, vertically centered against the paragraph's full height — a cramped, clearly-unintended two-column layout, screenshotted and reproduced twice. Root cause, traced in `style.css`: `.item-head { display:flex; align-items:center; flex-wrap:wrap }` combined with `.item-label { flex: 1 1 260px }` produces different flex-line breaks depending on exact slug/badge text width at this specific viewport, so otherwise-identical sibling cards lay out differently based on incidental content length.

### 8. Violations tab's empty state can't distinguish "never happened" from "happened and was resolved"

The Violations tab (correctly, deliberately mirrors the CLI's `led work violations` — current/unresolved only) shows "No violations — the decomposition tree currently has none unresolved" when empty. But this deployment's ledger holds six `work_violation_disposition` rows (confirmed via `/api/rows/facet-counts`) recording violations that *did* occur and were resolved, backed by a `work_violation_history` view (6 rows, confirmed via Backend Surface, "exposed by API: no," "dedicated tab: —"). A governance reviewer reading only the Violations tab has no way to learn this history exists or that it should be looked for elsewhere (`Recent ledger`, `kind: work_violation_disposition`). Given CLAUDE.md's own framing explicitly names "violations and their dispositions" as first-class domain content the SPA should make legible, a one-line pointer in the empty state (or a toggle to historical dispositions) would close this cheaply.

## MINOR

### 9. Review Gap tab shows raw principal ids, not names

`ReviewGapTab.vue`'s own header comment discloses this honestly: the underlying `review_gap` kernel view carries only `actor`/`assigned_by` as bare principal ids, with no join to a name — unlike every other tab in the app (Work items' `claimant_name`, Ledger's `actor_name`, Commission's `actor_name`), which all resolve to human-readable names. Currently unobservable live (`review_gap` has 0 rows in this deployment), but whenever it is non-empty a reader will see "actor 4 owes a review" rather than a name, inconsistent with the rest of the app's presentation.

### 10. Stale doc-comment pointing at a file that doesn't exist

`ReviewGapTab.vue`'s header comment references `src/views/stubs/ItemView.stub.vue` as the "intended home" for a full item-view route, justifying its own lightweight inline-expand-instead-of-navigate design. That file does not exist anywhere in the repository (confirmed by search) — a real, fully-built item view already lives at `/item/:id` (`src/ItemView.vue`). The comment predates that feature shipping and was never updated; its design rationale is now moot (the tab could link straight to the real item view instead of maintaining a separate fetch-and-expand-in-place mechanism).

### 11. Inconsistent depth of client-side row-id validation

`/item/abc` is caught client-side (`Number.isFinite` check) before any network call, showing a friendly "'abc' is not a valid row id" message with zero requests. `/item/-1` and `/item/3.5` pass that same check (both are finite numbers) and proceed to a real API round-trip that 404s, arriving at an equally friendly "no such row" message. The end state is identical and correct in both cases — this is cosmetic, not a functional defect — but the validation depth is inconsistent for no evident reason.

## What already meets a professional bar and should not regress

- **Zero unhandled errors across the entire surface tested.** Every tab, every filter interaction, every edge case (nonexistent row id, non-numeric id, negative id, decimal id, an unmatched route, an inverted date range) produced zero browser console errors, zero page errors, and zero *unexpected* network errors (the expected 404s on invalid ids are themselves evidence the app is handling failure correctly, not a defect).
- **Graceful 404 handling, both for bad row ids and bad routes**, with a real, reachable, branded in-app "Page not found" view rather than a blank screen — and confirmed working correctly on both direct navigation *and* full page reload (server-side history fallback is genuinely in place).
- **Real list virtualization** above 200 rows via `@tanstack/vue-virtual` (confirmed live at `limit=797` with superseded rows included — the app switches to a windowed `role="table"` div-grid rendering), addressing precisely the "unbounded append-only list" hazard `docs/omega-observatory/2026-07-15-frontend-speed-reap.md` warns about, including a previously-fixed row-height/overlap bug the code comments document by name.
- **A genuinely good responsive design below ~800px**: tables convert to legible, fully-labeled stacked cards (verified visually on Ledger and Work Items at 600px), the nav wraps cleanly onto two rows, and no tab I tested produced horizontal page overflow.
- **The hover-synopsis citation mechanism, where it is wired up, is well-built** — including a subtle, already-diagnosed-and-fixed bug (hover cards escaping a virtualized list's scroll-clipping ancestor via `Teleport`), and a live-verified working example (hovering a `row:` citation shows a synopsis card with statement/actor/age).
- **Read-only mode (`PANEL_READONLY`) is enforced and disclosed consistently everywhere** — every write control (co-sign buttons) I found across Commission, Item Detail, and Resource-field views was visibly disabled with an identical, correctly-worded explanation, with no dead-end or misleading "can I click this?" ambiguity anywhere.
- **The commission trust-level ladder (LAZY/FULL/SIGNED/FORGED-OR-CORRUPT/UNVERIFIABLE)** is rendered with well-calibrated badges and tooltip text that accurately reflects the nuance of `USER-GPG-TRUST-LAYER-FAQ.md`'s actual ladder (informational tone for the normal LAZY case, alarming tone reserved for the one genuinely alarming state) — a thoughtful domain-accurate touch, verified live on commission #741.
- **Composite-discharge legibility on Work Items is handled correctly and thoughtfully**: the tab distinguishes the kernel's raw `state` column from its derived `effective_state`, rendering `"open — done via children"` for a composite parent discharged by its children rather than silently showing a misleading raw `"open"` — directly addressing the exact hazard `Autoharn.idr` §4 is devoted to, and confirmed live on `spa-audit-cycle-4-fixes`.
- **Specific, correctly-worded empty states throughout** (including a dedicated message for the `limit=0` edge case, distinct from a generic "no rows"), rather than a blank table that could read as broken.
- **The kind-filter vocabulary is genuinely live-derived** from the backend's own facet-counts rather than hand-copied — verified that the absence of a "revision" kind option is because zero rows of that kind exist today, not a hardcoded omission, so the dropdown cannot silently drift stale as new kinds are introduced.
