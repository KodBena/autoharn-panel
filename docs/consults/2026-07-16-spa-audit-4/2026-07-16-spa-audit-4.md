# Independent Audit: autoharn-panel SPA vs. an SAP/Oracle-Class Enterprise Bar

**Auditor**: independent, single-pass, live-driven (Playwright/headless Chromium via Bash, plus direct API/Postgres introspection for ground truth). No prior findings were consulted; this audit treats itself as the only one on record.

**Target examined**: the live deployment described by `/home/bork/w/vdc/1/experience/autoharn-panel/deployment.json` (`db=toy`, `schema=experience`/`experience_kernel`, `role=experience_rw`, host `192.168.122.1`), reached through the actual running FastAPI backend at `http://127.0.0.1:8420` (PID 2038956, `PANEL_READONLY=locked`) — this is the "real" serving deployment and is what this report treats as *the* live app. A Vite dev server was also found running at `127.0.0.1:5183` (proxying to the same backend); it was used only as a **diagnostic control** to distinguish "current source code behaves correctly" from "the artifact actually being served does not" — see Finding 3, which depends on that distinction.

**Method**: read `autoharn/README.md`, `autoharn/docs/PROJECT-OVERVIEW.md`, `autoharn/design/USER-RECIPES-FAQ.md`, and `autoharn/design/Autoharn.idr` in full first, to understand the ledger's actual domain (commissions and their LAZY/FULL/SIGNED trust strengths; work items with open/claim/depends/close, typed dependency edges distinct from parent/child, and composite auto-discharge; reviews and review-gap obligations; graded/standing durable decisions; questions, assumptions, findings, snags, notes as symmetric prose kinds; violations and their dispositions; resource/taxonomy declarations). That understanding is the lens applied below — a capability the ledger genuinely has and the SPA has no view for is treated as exactly as real a defect as a rendering bug, and the two are ranked together, not separated. Live exploration used headless Chromium driven by scripted Playwright through Bash (never the Playwright MCP tool); direct `psql` introspection against the live Postgres instance (schema/view/column shapes and row *counts* only, never reading ledger prose content for its own sake) was used to establish ground truth for what data genuinely exists, independent of what the app chooses to show.

## Findings, ranked by real-world severity/impact

### 1. CRITICAL — Composite work items that are actually done are displayed as plain "open," indistinguishable from stalled work

The kernel's own derived view, `experience.work_item_current`, carries an `effective_state` column implementing exactly the composite-discharge semantics `Autoharn.idr` §4 documents at length (a composite parent auto-discharges once its children's obligation tree resolves — "no manual close needed"). Live query:

```
slug                         | state | effective_state           | parent_slug
spa-professionalism-cycle-1  | open  | discharged-by-obligations |
spa-professionalism-cycle-3  | open  | discharged-by-obligations |
```

Both items are, by the kernel's own authoritative derivation, **fully complete**. But `GET /api/work` (and the Work Items tab built on it) returns only the raw columns:

```json
{"slug": "spa-professionalism-cycle-1", "state": "open", "resolution": null, "witness": null, "claimant_name": null}
```

Verified live in the browser (Work Items tab, filtered to `spa-professionalism-cycle-1`): STATE column reads `open`, CLAIMANT and RESOLUTION are blank. The item's own title text says, verbatim, *"discharge is composite (s33), auto-resolves when children close, no manual close needed"* — the SPA does not implement the semantic its own ledger content describes to the reader. Neither `/api/work`, `/api/rows/{id}`, nor `/api/item/{id}/obligations` ever selects `effective_state` or `parent_slug` anywhere. A reader has no way to tell "genuinely stalled" from "done via children" for any composite item — a governance dashboard silently misrepresenting the completion state of real, current work is a first-order trust failure.

### 2. CRITICAL — Commission Decomposition shows "0 item(s)" for 87% of real commissions with substantial completed work

`GET /api/commissions` against the live deployment:

```
total commissions: 15
commissions with item_count > 0: 2   (row 434: 2 items, row 514: 1 item)
```

The other 13 — including commission `#48` ("do 1 and 2 right away... decompose... review and countersign"), `#183`, `#247`, `#374` and more — all report **`item_count: 0`**, confirmed directly in the running UI's own dropdown text: *"#48 — 0 item(s)"*, *"#183 — 0 item(s)"*, etc. This is false: the Work Items tab shows dozens of `closed`/`shipped` items whose titles explicitly read `--refs row:48`, `--refs row:247`, `--refs row:183`. The root cause is already self-diagnosed inside this very ledger (a finding recorded while investigating `fix-commission-decomposition-view`): the `led work open` CLI grammar never actually writes a `refs` value onto a `work_opened` row, so the SPA's join from commission → decomposed items structurally cannot find them for any but a couple of specially-handled recent commissions. The Commission Decomposition tab — one of only six top-level views, and the single most direct way to answer "what did we actually decompose and ship against this ask" — is silently, systematically wrong for the overwhelming majority of this deployment's real history. It fails silently: no warning, no "this view can't link most historical items," just a plausible-looking `0`.

### 3. CRITICAL — The live production frontend is a stale build: broken deep-linking, missing 404 page, missing ARIA roles, and a reproduced garbled-table rendering bug, all from one root cause

Direct, reproducible comparison between the actual serving artifact (`frontend/dist`, built **2026-07-16 07:11**) and current frontend source (`src/router.ts` / `src/App.vue`, last modified **2026-07-16 16:38** — over 9 hours later) shows the deployed bundle predates logic present in source right now. Confirmed live, side-by-side:

| Behavior | Prod build (`:8420`, what real users get) | Dev server (fresh source, `:5183`) |
|---|---|---|
| Cold direct nav to `/review-gap`, `/work-items`, `/questions`, `/commissions`, `/profiles` | **All 5 fall back to "Recent ledger"** — the URL bar shows the right path, the rendered tab is wrong, every time | Correctly activates the requested tab |
| Nav to an unrecognized path (`/totally/bogus/path`) | Silently renders "Recent ledger" | Renders a real, distinct "Page not found" view |
| Tab bar ARIA roles | None found (`role=status` only) | `role=tablist` + `role=tab` on every tab button, `aria-selected` wired |
| Recent Ledger at `limit=700` | Rows visibly **overlap and interleave text**, unreadable (screenshotted) | (not retested at this limit, but the source-level fix for this exact bug is present) |

Every one of these is a *known, previously-identified defect* (a matching self-diagnostic already exists in this deployment's own history for each) that current source appears to address, but the artifact actually being served to anyone visiting the deployment right now regresses all four simultaneously. This is one root cause (an unrebuilt/unrestarted frontend artifact), not four independent bugs, and the fix is a single rebuild-and-redeploy step — but until that happens, **every real visitor bookmarking, sharing, or reloading any of 5 of the app's 6 tabs lands on the wrong screen**, and anyone raising the Recent Ledger row limit past ~200 gets an unreadable table. Ranked CRITICAL for breadth (hits every tab, every user) and for including an outright-unusable rendering failure, not merely an inconvenience.

### 4. SERIOUS — Standing/durable decisions (a real, kernel-supported concept) are completely invisible

`Autoharn.idr`'s header names s36 ("decision-grade... `standing_decisions` in-force view") as landed governance semantics, and this is real, not aspirational, in the live deployment: `experience.standing_decisions` is a live view with **22 rows**, all graded `'durable'`. Neither `GET /api/rows/{id}` nor any other endpoint ever selects a grade/durable field — confirmed directly (`/api/rows/397`, a row the kernel itself marks durable, returns `kind`/`statement`/`ts`/`refs`/`supersedes`/`predecessors`/`successor` and nothing else). The SPA has no way to distinguish a decision meant to stand as authoritative guidance surviving context loss from an ordinary one-off note — all 184 `decision`-kind rows in Recent Ledger render identically regardless of what the kernel itself has flagged.

### 5. SERIOUS — Work-item dependency edges and parent/child structure are structurally absent from every layer of the app

Beyond the concrete instance in Finding 1, the gap is total and architectural. The kernel exposes real typed data — `work_edge_blocks_close` (2 live edges, e.g. `cycle3-unknown-path-404` blocks-close-depends-on `cycle3-tab-url-routing`) and `work_edge_parent` (26 live edges) — plus `work_item_current.parent_slug`. **None of it reaches `/api/work`, `/api/rows/{id}`, or `/api/item/{id}/obligations`.** There is no dependency graph view, no composite/parent tree visualization, nowhere in the app. A reader can only infer composite structure by parsing free-text "Parent: ..." prose inside a work item's title — exactly the "typed edge vs. hand-parsed prose" distinction `Autoharn.idr` itself treats as load-bearing (`EdgeType = BlocksClose | Informs`, deliberately typed and refused-if-cyclic at the kernel level) is invisible at the presentation layer.

### 6. SERIOUS — Resource declarations get no structured display; the whole tier vocabulary is opaque prose

Two live `resource:` rows exist (e.g. `makespan-scheduler`, tier `blessed`). Visited directly (`/item/397`), the item detail page shows the entire six-field pipe-delimited convention (`name | class | reach | what-it-proves | guidance | tier`) as one undifferentiated blob of statement text — no parsed tier badge, no dedicated "Resources" table, nothing distinguishing `available`/`blessed`/`mandated`/`forbidden`. The operator's own CLI (`./pickup`) already renders this tier-sorted with prohibitions first; the SPA, whose whole purpose is to be the polished reflection of this same ledger, does not. (No `taxon:`/`interface:` rows exist yet in this deployment, so there is nothing concrete being hidden there today — but there is equally no scaffolding, not even an empty state, for the concept.)

### 7. MODERATE — Commission trust/attribution strength is never parsed or surfaced

Every commission in this deployment literally begins `(vicarious transcription by the implementer; carries no commissioner guarantee)` — the LAZY-mode marker `CLAUDE.md`/`USER-GPG-TRUST-LAYER-FAQ.md` define as one of three meaningfully different trust strengths (LAZY/FULL/SIGNED), where a CONTESTED commission *must* be SIGNED to stand. `/api/commissions` and `/api/commission/{id}` pass this through as raw prose only — no trust-level badge, no `verify-commission` verdict (VERIFIED/UNSIGNED/FORGED-OR-CORRUPT) surfaced anywhere. A reader must notice and parse the sentence themselves to know how much to trust a commission's attribution.

### 8. MODERATE — Assumptions/findings/snags/notes get no treatment beyond generic kind-filtering

`Autoharn.idr` models all eight prose kinds (assumption, decision, question, verification, finding, snag, revision, note) symmetrically as one `ProseKind` sum type. The SPA treats two of them as first-class (Questions gets a dedicated tab with a computed answered/unanswered projection; Review gap gets a dedicated computed obligation queue) and the rest — including findings and snags, which are exactly the kind of recorded-defect/observation content an audit-conscious reader most wants surfaced distinctly — get nothing beyond the same `kind:` dropdown filter every other row type shares. There is no "is this finding still open," no cross-referencing, no distinct visual treatment at all.

### 9. MODERATE — No hover-synopsis on citation links

`row:N`-style citations throughout the app (Recent Ledger, Work Items, Commission Decomposition, Review gap, Questions) are clickable links to `/item/N`, but hovering produces no preview/synopsis of the target content — confirmed absent both on the production build and on fresh dev-server source, so this is a standing gap, not a build-staleness symptom. In a UI this densely cross-referenced, a reader must fully navigate away and back just to check what a citation says.

### 10. MODERATE — Violation dispositions have no dedicated queue and aren't auto-linked

`work_violation_disposition` rows (6 historical, e.g. `orphaned_by_retraction target 602 (retired)`) are reachable only via the generic `kind:` filter in Recent Ledger — there is no dedicated "currently in-force violations" view mirroring the treatment Review gap already gets (the kernel's own `work_item_violations` view is exactly that "what's currently wrong" signal; it happens to be empty right now, but nothing in the app would surface it if it weren't). The disposition text's own target reference ("target 602") is plain text, not a resolved `row:602` citation link, despite the app's citation-linking convention existing and being applied everywhere else.

### 11. MINOR — Questions tab shows no inline question text

The Questions tab's table (QUESTION ROW / KIND / ANSWERED / FIRST ANSWER ROW / ANSWERS NON-QUESTION TARGET) never shows a snippet of the actual question — a reader must click through to `/item/N` to learn what was even asked, unlike Recent Ledger's own truncate-with-expand-in-place pattern for long statements.

### 12. MINOR — SSE "live" badge has no heartbeat independent of real ledger activity

`/api/events`'s generator (`backend/core/routes.py`) only ever `yield`s when a genuine ledger write occurs — confirmed live (12+ seconds, zero bytes beyond headers, on an idle deployment). The frontend's status flips to `live (SSE)` on the browser's `EventSource` `open` event (confirmed working correctly against the actual origin, `:8420`), so this is not currently broken — but there is no periodic proof-of-liveness independent of new writes, so a genuinely stalled connection and a merely quiet one would be indistinguishable to the badge between two real ledger writes.

### 13. MINOR — Benign console noise on invalid item IDs

Visiting `/item/999999999` correctly renders a clean "no such row" empty state, but the underlying failed fetch still logs a `Failed to load resource: 404` line to devtools. Cosmetic only, not user-visible, but a strict zero-console-error bar would flag it.

## What already meets a professional bar — should not regress

- **Read-only/locked-deployment messaging** is honest and layered: a header badge with an explanatory tooltip, and disabled write actions (co-sign buttons) that explain *why* they're disabled in the surrounding text rather than surfacing a bare, indistinguishable-from-broken HTTP 405.
- **Recent Ledger** is a genuinely capable table: sortable columns (verified — clicking the ID header re-sorts ascending), `kind`/`actor`/date-range facets sourced live from real facet-counts, working `Prev`/page-N/`Next` pagination, and a truncate-with-expand-in-place pattern for long statements that keeps the default view scannable.
- **The deep-linkable item view** (`/item/<id>`) is well-built where it works: kind/actor/timestamp/statement, resolved `refs`, a rendered supersede chain (predecessors/successor), and a reviews/witnesses/co-sign section in one place.
- **Invalid-ID handling** is graceful: `/item/999999999` shows a clear "no such row" message rather than a crash or a raw error.
- **Inline vocabulary glossary tooltips** (ADR numbers, `sNN` kernel-generation identifiers) resolve jargon for a cold reader inline — a genuinely nice, non-obvious polish touch.
- **Responsive layout at ~600px** holds up: no horizontal page overflow, tabs and the Commission Decomposition view reflow acceptably.
- **Backend input validation** is solid where it's been applied: `limit=-5` and `limit=0` on `/api/rows` both correctly return HTTP 400 with a field-level message rather than a 500 or silent misbehavior.
- **Server-side history-fallback** genuinely works: every path tested (`/review-gap`, `/totally/bogus/path`, `/item/1`) returns HTTP 200 with the SPA shell from the server — the routing failures in Finding 3 are a client-side, stale-bundle problem, not a missing server-side fallback.
